# Marts data-model reference with snowflake graphs

Tracking issue: [#4120](https://github.com/TEAMSchools/teamster/issues/4120)

## Goal

A generated MkDocs reference page documenting every `kipptaf` fact table, each
with a Mermaid ER snowflake diagram (full FK chain) and a foreign-key table.

## Background

The `kipptaf` marts layer (`src/dbt/kipptaf/models/marts/`) holds 24 fact
tables, ~42 dimensions, and 8 bridges, all contract-enforced views. The design
is a strict-chain snowflake: facts FK to their direct parents only, and deeper
dimensional context is reached by chaining `dim -> parent dim` (e.g.
`fct_student_attendance_daily -> dim_student_enrollments -> dim_students`). FK
edges are declared in each model's properties YAML as column-level
`constraints: { type: foreign_key, to: ref('dim_x') }`. Mermaid is not yet
enabled in the MkDocs `superfences` config.

## Scope

- **In:** a generator script, the generated reference page, `mkdocs.yml` wiring
  (mermaid fence + nav entry).
- **Out:** grain/description/measures text per fact, a dimension/bridge index,
  CI drift enforcement. (Each fact section shows only the ER diagram + FK
  table.)

## Components

### 1. Generator script — `scripts/generate_marts_reference.py`

- Run via `uv run python scripts/generate_marts_reference.py`. No dbt build or
  warehouse access — parses YAML directly with PyYAML so it runs anywhere.
- Reads
  `src/dbt/kipptaf/models/marts/{facts,dimensions,bridges}/properties/*.yml`.
- Builds a directed FK graph. For every column carrying a `foreign_key`
  constraint, add an edge `model --(fk_column)--> target`, where `target` is
  regex-extracted from the `ref('...')` string. Nodes are facts **and** dims
  (fact->fact edges exist for 4 facts and must be traversed).
- For each `fct_*`, BFS over outgoing edges to assemble the **full snowflake
  chain** (dims -> parent dims, plus parent-fact edges). A visited set keeps
  shared dims (e.g. `dim_dates`) to a single node per fact subgraph.
- Role-qualified FKs to the same target (e.g. `submitter_staff_key`,
  `assignee_staff_key`, `original_assignee_staff_key` -> `dim_staff`) render as
  distinct labeled edges; the target node appears once.
- Emits one markdown file beginning with
  `<!-- generated by scripts/generate_marts_reference.py; do not edit by hand -->`.

### 2. Output page — `docs/reference/marts-data-models.md`

- Intro: the strict-chain snowflake design + a legend for reading the ER
  diagrams.
- Per fact (`##` heading, alphabetical): a Mermaid `erDiagram` of its full
  snowflake chain, then an FK table with columns `FK column | -> target model`.
- Material auto-generates the page TOC from the `##` headings.

### 3. MkDocs wiring — `mkdocs.yml`

- Convert the bare `- pymdownx.superfences` to the mapping form with a `mermaid`
  custom fence (standard Material setup) so ` ```mermaid ` blocks render.
- Add `- Marts Data Models: reference/marts-data-models.md` under the
  `Reference` nav section.

### 4. Regeneration

- Documented in the page banner and a one-line note in `scripts/CLAUDE.md`:
  `uv run python scripts/generate_marts_reference.py`.

## Data flow

`marts properties YAML` -> `generate_marts_reference.py` (parse + graph + BFS)
-> `docs/reference/marts-data-models.md` -> MkDocs build (mermaid renders
client-side).

## Edge cases

- Fact->fact edges (parent-fact inheritance) — traverse as normal nodes.
- Role-qualified multiple FKs to one target — one labeled edge each, single
  target node.
- DAG assumption (no diamonds by design) — visited set guards against
  re-emitting a shared dim within one fact's subgraph.
- A fact whose direct dims fan out to many parents produces a large diagram;
  acceptable for v1 (auto-TOC keeps the page navigable).

## Testing / verification

- Run the generator; confirm all 24 facts produce a section.
- `uv run mkdocs build --strict` succeeds and mermaid fences render (spot-check
  the built HTML contains `class="mermaid"` blocks).
- Spot-check 2-3 facts against their YAML: every `foreign_key` constraint
  appears as both an edge and an FK-table row.
